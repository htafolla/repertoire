## Thread (reply to primary)

### 2/7 — the problem

Most agent stacks route every task the same way.

Same tier. Same agent. No memory of what burned you last time.

Repertoire fixes that — it reads curated signals from real inference runs and tells the orchestrator *when* to escalate, *who* to call, and *why*.

### 3/7 — trap routing

The sharp edge: **ontological traps**.

Tasks that *look* simple but need architect + security review — not a fast codegen pass.

`repertoire__get_task_confidence` catches them, boosts complexity, and routes accordingly.

Wired into `xray-researcher` as auditable `MEMORY_ROUTING:` blocks. 🎯

### 4/7 — MCP (external hosts)

Running Grok · Hermes · OpenCode · OpenClaw?

Add one server to `.mcp.json`:

```json
"repertoire": {
  "command": "npx",
  "args": ["-y", "@0xray/repertoire", "mcp"]
}
```

4 tools: confidence · search · high-confidence signals · feedback ingest 🔌

### 5/7 — in-process (0xRay orchestrator)

For in-process routing, flip it on in `features.json`:

```json
"memory_routing": {
  "enabled": true,
  "provider": "repertoire"
}
```

Enriches `ExecutionPlanner`, `thinDispatch`, and `AsideContext` — not just researcher votes. 🧠

### 6/7 — the loop

Repertoire isn't read-only.

`repertoire__ingest_feedback` records orchestrator outcomes back to the registry.

Ingest Groover inference logs → meta-inference → `curated_signals.json` → smarter routing next run.

Memory that compounds. ♻️

### 7/7 — get started

```bash
npm install @0xray/repertoire
```
